## 一、方案选型对比

### 1.1 主流微前端方案矩阵

| 维度           | micro-app                  | qiankun                   | wujie                      | Module Federation         | iframe          |
| -------------- | -------------------------- | ------------------------- | -------------------------- | ------------------------- | --------------- |
| **技术原理**   | WebComponent + 沙箱        | single-spa + import-html  | WebComponent + iframe 沙箱 | Webpack/Vite 模块共享     | 浏览器原生隔离  |
| **接入成本**   | ⭐ 极低（类组件标签）      | ⭐⭐ 中等（生命周期改造） | ⭐ 低                      | ⭐⭐⭐ 高（构建配置）     | ⭐ 极低         |
| **JS 沙箱**    | ✅ Proxy + iframe 双模式   | ✅ Proxy（单实例限制）    | ✅ iframe 天然隔离         | ❌ 无沙箱                 | ✅ 天然隔离     |
| **CSS 隔离**   | ✅ 自动 scoped             | ⚠️ 需手动处理             | ✅ iframe 天然隔离         | ❌ 无隔离                 | ✅ 天然隔离     |
| **通信机制**   | 数据绑定 + PostMessage     | Actions + Props           | Props + PostMessage        | 共享模块直接调用          | PostMessage     |
| **预加载**     | ✅ `microApp.preFetch()`   | ✅ 支持                   | ✅ 支持                    | ✅ 天然支持               | ❌ 不支持       |
| **Vite 兼容**  | ✅ 原生支持（iframe 沙箱） | ⚠️ 需 vite-plugin-qiankun | ✅ 原生支持                | ✅ vite-plugin-federation | ✅              |
| **多实例**     | ✅ 支持                    | ❌ 不支持                 | ✅ 支持                    | ✅ 支持                   | ✅ 支持         |
| **Keep-Alive** | ✅ 原生属性支持            | ❌ 需自实现               | ✅ 原生支持                | ❌ 不适用                 | ⚠️ display:none |
| **维护状态**   | 🟢 活跃（京东开源）        | 🟡 低频（蚂蚁）           | 🟢 活跃（腾讯）            | 🟢 活跃（Webpack）        | —               |
| **包体积**     | ~15KB                      | ~30KB                     | ~20KB                      | 0（构建时）               | 0               |

### 1.2 为什么选择 micro-app

| 项目需求              | micro-app 方案                        | 匹配度  |
| --------------------- | ------------------------------------- | ------- |
| Vue 3 + Vite 8 技术栈 | 原生支持，无需额外插件                | ✅ 完美 |
| 类 HTML 标签式开发    | `<micro-app>` 标签                    | ✅ 完美 |
| 子应用技术栈无关      | 支持 Vue/React/Angular/jQuery         | ✅ 完美 |
| 主题统一同步          | `data` 绑定 + PostMessage + `setData` | ✅ 完美 |
| 子应用路由保活        | `keep-alive` 属性                     | ✅ 完美 |
| 渐进式接入            | 子应用零改造                          | ✅ 完美 |
| iframe 沙箱模式       | Proxy + iframe 双沙箱                 | ✅ 完美 |
| 预加载                | `microApp.preFetch()`                 | ✅ 完美 |

### 1.3 已知限制与缓解方案

| 限制                             | 影响                         | 缓解方案                            |
| -------------------------------- | ---------------------------- | ----------------------------------- |
| `<micro-app>` 标签名需全小写     | ESLint PascalCase 规则冲突   | `eslint-disable` 单行注释           |
| iframe 沙箱性能略低于 Proxy 沙箱 | 首次加载多 ~50-100ms         | 预加载 `preFetch` + keep-alive 缓存 |
| 子应用路由需统一前缀             | 路由命名空间约束             | 规范化 `baseroute` 配置             |
| 跨域需 CORS 配置                 | 开发环境跨端口调试           | Vite proxy / `cors: true`           |
| iframe 沙箱中 `history` 隔离     | 子应用无法直接操作主应用路由 | PostMessage 通知主应用跳转          |

### 2.2 技术选型

| 层面       | 技术                 | 版本         | 说明              |
| ---------- | -------------------- | ------------ | ----------------- |
| 微前端框架 | @micro-zoe/micro-app | ^1.0.0-rc.29 | 京东开源          |
| 主应用框架 | Vue 3 + Vite 8       | 3.5 / 8.x    | 渐进式框架        |
| UI 组件库  | Naive UI             | 2.41+        | 全量 TypeScript   |
| 状态管理   | Pinia + 持久化       | 3.0          | Setup Store       |
| 沙箱模式   | iframe 沙箱          | —            | Vite 推荐模式     |
| 主题系统   | @robot-admin/theme   | —            | Dark/Light/System |

### 2.3 核心设计原则

1. **零改造接入** — 子应用不需要暴露 `bootstrap/mount/unmount`，micro-app 自动渲染
2. **类组件开发** — `<micro-app name="app" url="..." />` 像 Vue 组件一样使用
3. **门户工作台** — 统一的系统入口，三栏布局聚合所有业务系统
4. **主题统一** — 主应用 Dark/Light 模式实时同步到所有子应用
5. **渐进式演进** — 从单体 → Monorepo → 微前端，平滑升级

### 2.4 应用启动流程

```
main.ts bootstrap()
  │
  ├─ setupLoading()              # 0. 首屏加载动画
  ├─ setupMicroApp()             # 1. micro-app.start() ← 必须在 createApp 前
  ├─ createApp(App)              # 2. 创建 Vue 实例
  ├─ setupStore(app)             # 3. Pinia + 持久化
  ├─ setupRequestCore(app)       # 4. Axios 请求核心
  ├─ setupLayoutSystem(app)      # 5. 布局系统
  ├─ setupNaiveUI(app)           # 6. Naive UI 全局通知
  ├─ setupDirectives(app)        # 7. 自定义指令
  ├─ router.isReady()            # 8. 等待路由就绪
  └─ app.mount('#app')           # 9. 挂载
```

> **关键**：`setupMicroApp()` 必须在 `createApp()` 之前调用，确保 `<micro-app>` 自定义元素在 Vue 编译前注册。

## 三、目录结构规范

> 对齐 micro-app 官方社区最佳实践的四层结构，清晰分离主应用、共享契约层、子应用和文档。

### 3.1 官方社区最佳实践目录结构

> 基于 micro-app 官方文档 + 京东/腾讯等大厂微前端落地经验总结。

```
my-platform/
├── main-app/                     # 主应用（统一入口、鉴权、路由）
│   ├── src/
│   │   ├── config/               # 子应用注册表（地址/环境/元信息）
│   │   ├── plugins/              # micro-app.start() 插件
│   │   ├── views/
│   │   │   ├── portal/           # 门户工作台（系统聚合入口）
│   │   │   └── micro-container/  # 微应用容器页（<micro-app> 渲染）
│   │   └── types/                # Window 扩展 + 通信类型声明
│   └── package.json
│
├── sub-apps/                     # 所有子应用（可独立仓库）
│   ├── sub-app-a/                # 子应用 A（任意技术栈）
│   ├── sub-app-b/                # 子应用 B
│   └── sub-app-c/                # 子应用 C
│
├── shared/                       # 主子应用共享包（关键！）
│   ├── types/                    # 通信数据接口类型定义
│   ├── constants/                # PostMessage 消息类型常量枚举
│   └── utils/                    # 共享工具函数（格式化/鉴权工具等）
│
└── docs/                         # 架构文档
    ├── architecture.md           # 架构总览
    ├── communication.md          # 通信协议约定（提供给子应用团队）
    └── onboarding.md             # 子应用接入规范
```

**官方核心建议**：

- `shared/` 包是**微前端协作的关键契约层**，主应用团队维护，子应用团队消费
- 子应用应尽量保持**技术无关性**，不依赖主应用的内部模块
- 通信协议（消息类型 + payload 格式）应文档化并版本化
- 子应用配置表（注册表）建议提取到独立文件，方便未来改为**后端动态配置**

```
Robot_Admin/
├── src/                    # 主应用（Robot Admin 系统本身）
```

### 3.3 `shared/` 契约层

主应用团队维护，所有子应用通过 `@shared/*` 路径别名消费：

```typescript
// 主应用 tsconfig/tsconfig.app.json
"paths": { "@shared/*": ["shared/*"] }

// 子应用 micro-apps/logistics/tsconfig.json
"paths": { "@shared/*": ["../../shared/*"] }
```

当前契约内容：

| 文件                        | 内容                                                 |
| --------------------------- | ---------------------------------------------------- |
| `shared/constants/index.ts` | 12 种消息类型 + 2 个自定义事件名 + 1 个存储键        |
| `shared/types/index.ts`     | `MicroAppData` / `MicroAppMessage<T>` / 5 种 Payload |
| `shared/utils/index.ts`     | `createMessage()` / `parseMessage()` 工具函数        |

### 3.4 `micro-apps/` 子应用目录

每个子目录是**独立完整的项目**（独立 `package.json` / `vite.config.ts` / `tsconfig.json`）：

- 开发阶段：在 `micro-apps/` 本地联调，主子应用协同开发
- 生产阶段：子应用**独立部署**到各自服务器，主应用通过 URL 加载
- 演进阶段：当团队规模扩大，子应用可随时抽离为**独立 Git 仓库**

### 3.5 各目录职责表

| 目录                      | 职责                           | 维护主体                       | 扩展方式                           |
| ------------------------- | ------------------------------ | ------------------------------ | ---------------------------------- |
| `src/`                    | 主应用全量业务代码             | 主应用团队                     | 按功能模块新增子目录               |
| `src/config/microApps.ts` | 子应用注册表（地址+环境映射）  | 主应用团队                     | 新增子应用时追加 `MICRO_APPS` 条目 |

| `shared/`                 | 主子通信契约（类型/常量/工具） | 主应用团队维护，子应用团队消费 | 新增消息类型时在 `constants/` 扩展 |
| `docs/`                   | 架构文档 + 接入规范            | 主应用团队                     | 提供子应用接入说明                 |

```
## 四、核心模块分析

> 本章从架构角度分析各核心模块的职责和设计决策，具体代码实现详见 [README — 集成方式](../README.md#-集成方式)。

### 4.1 子应用配置表 — `microApps.ts`

**设计决策**：采用静态配置表 + 多环境地址映射模式。

| 设计选项             | 当前方案           | 分析                                                             |
| -------------------- | ------------------ | ---------------------------------------------------------------- |
| 配置存储             | 本地 TS 文件       | 简单直接，适合早期；未来可迁移到后端 API 实现动态注册            |
| 环境地址             | dev/test/staging/prod 四级 | 覆盖完整发布流程，`getMicroAppUrl()` 自动根据 `import.meta.env.MODE` 选择 |
| 元信息（icon/desc）  | 直接嵌入配置       | 驱动门户页 UI 渲染，避免硬编码散落；未来可拆分为独立 registry    |

**改进方向**：增加 `preFetch` / `keepAlive` / `timeout` / `requiredRoles` 等运行时控制字段。

### 4.2 微前端启动插件 — `micro-app.ts`

**当前配置**：仅启用 `disable-memory-router: false` + `disable-patch-request: false`（最小化配置）。

**官方可用配置项分析**：

| 配置项                  | 是否启用 | 分析                                                   |
| ----------------------- | -------- | ------------------------------------------------------ |
| `iframe`                | 全局未开 | 在 `<micro-app>` 标签上按需开启，更灵活                |
| `fiber`                 | ⬜       | 重型子应用需要，建议 Phase 2 按需开启                  |
| `globalAssets`          | ⬜       | 多子应用共享 Vue/Naive UI 时收益大，建议 Phase 2       |
| `inline`                | ⬜       | 仅调试用，生产不要开                                   |
| `destroy`               | ⬜       | 低频子应用建议开启以释放内存                           |

### 4.3 微应用容器 — `micro-app/index.vue`

**架构角色**：统一渲染入口，所有子应用复用同一个容器页，通过路由参数 `:appId` 动态切换。

**核心能力**：

| 能力             | 实现方式                          | 分析                                       |
| ---------------- | --------------------------------- | ------------------------------------------ |
| 动态加载         | `:name="appId"` + `:url="appUrl"` | 支持 N 个子应用共用一个容器，路由驱动      |
| 数据注入         | `:data="appData"` computed        | 响应式推送 token/主题/环境/工具函数        |
| 事件监听         | `@datachange` + PostMessage       | 6 种消息类型，覆盖路由/数据/生命周期/自定义 |
| 主题同步         | `watch(themeStore.mode)`          | setData 精准推送，不重渲染                 |
| 沙箱隔离         | `iframe` 属性                     | Vite ESM 必须用 iframe 沙箱               |
| 状态保活         | `keep-alive` 属性                 | 切换不销毁，恢复瞬间                       |

### 4.4 门户页 — `portal/index.vue`

**架构角色**：系统聚合入口（仪表盘模式），三栏布局。

| 区域   | 内容                             | 作用                           |
| ------ | -------------------------------- | ------------------------------ |
| 左栏   | 用户卡片 + 待办 + 外部链接      | 个人工作台                     |
| 中栏   | 子应用数据推送区 + 快捷入口     | 聚合子应用核心指标             |
| 右栏   | 天气（wttr.in）+ 日历           | 辅助工具                       |

## 五、通信架构设计

> 具体代码示例详见 [README — 数据通信](../README.md#-数据通信)。

### 5.1 通信架构全景

```

                   ┌──────────────────────────────┐
                   │         micro-app 框架层       │
                   │   (代理 data / dispatch / 事件) │
                   └────────────┬─────────────────┘
                                │
         ┌──────────────────────┼──────────────────────┐
         │                      │                      │
    ① data 属性绑定         ② setData/getData       ③ PostMessage
    (主 → 子, 响应式)       (命令式, 双向)          (子 → 主, 跨 iframe)
         │                      │                      │
         ▼                      ▼                      ▼
    初始化数据注入        运行时精准推送          事件通知 + 路由跳转

```

### 5.2 四种通信方式对比

| 通信方式         | 方向    | 触发机制 | 适用场景                       | 是否已实现 |
| ---------------- | ------- | -------- | ------------------------------ | ---------- |
| `data` 属性绑定  | 主 → 子 | 响应式   | 初始化注入（token/主题/环境）  | ✅         |
| `setData/getData` | 双向   | 命令式   | 运行时动态推送（主题切换等）   | ✅         |
| PostMessage      | 子 → 主 | 事件     | 路由跳转、自定义业务消息       | ✅ 6 种    |
| `globalData`     | 广播   | 一对多   | 多子应用间共享数据             | ⬜ 待实现  |

### 5.3 已实现的 6 种消息类型

| 消息类型             | 方向    | 用途                   | 定义位置                         |
| -------------------- | ------- | ---------------------- | -------------------------------- |
| `MICRO_APP_NAVIGATE` | 子 → 主 | 子应用请求主应用跳转   | `@shared/constants`              |
| `CUSTOM_MESSAGE`     | 子 → 主 | 自定义业务消息         | `@shared/constants`              |
| `DATA_UPDATE`        | 子 → 主 | 子应用推送数据到门户   | `@shared/constants`              |
| `MOUNTED`            | 子 → 主 | 子应用挂载完成通知     | `@shared/constants`              |
| `ROUTE_CHANGE`       | 子 → 主 | 子应用路由变化通知     | `@shared/constants`              |
| 主题同步             | 主 → 子 | setData 推送暗色模式   | `micro-app/index.vue` watch      |

### 5.4 通信场景决策矩阵

| 场景                          | 推荐方式        | 原因                      |
| ----------------------------- | --------------- | ------------------------- |
| 初始化注入（token/主题/环境） | `data` 属性绑定 | 响应式，自动同步          |
| 主题切换实时同步              | `setData`       | 精准推送，不重新渲染      |
| 子应用请求路由跳转            | PostMessage     | iframe 沙箱隔离了 history |
| 子应用推送业务数据            | PostMessage     | 异步，不阻塞渲染          |
| 多子应用间共享数据            | globalData      | 广播机制，一对多          |
| 子应用挂载完成通知            | `@mounted` 事件 | 框架级生命周期回调        |

### 5.5 通信安全分析

**当前风险**：PostMessage 使用 `'*'` 通配符，生产环境需限制 origin。

**改进方案**：

```

子应用发送: window.parent.postMessage(data, 'https://admin.example.com') ← 限定 origin
主应用验证: event.origin ∈ getAllMicroApps().map(app → getMicroAppUrl(app.id))

````

> 详见 [十、现状差距 — PostMessage 安全增强](#104-postmessage-安全增强)。

## 六、沙箱与隔离策略

### 6.1 沙箱模式决策

| 模式                  | 原理                       | 优势     | 劣势           | 适用场景              |
| --------------------- | -------------------------- | -------- | -------------- | --------------------- |
| **with 沙箱**（默认） | Proxy 代理 window          | 性能较高 | `var` 声明泄漏 | 非 Vite 项目          |
| **iframe 沙箱**       | 创建 iframe 隔离 JS 上下文 | 完全隔离 | 性能略低 ~50ms | **Vite 项目（推荐）** |

**本项目决策**：使用 **iframe 沙箱**，因为 Vite 8 的 ESM 模块在 with 沙箱下存在兼容问题。

### 6.2 样式隔离分析

| 隔离策略       | 原理                   | 适用沙箱       | 本项目状态                 |
| -------------- | ---------------------- | -------------- | -------------------------- |
| scopecss       | micro-app 自动加前缀   | with 沙箱      | 不适用（使用 iframe）      |
| iframe 天然隔离 | iframe 内外 CSS 互不影响 | iframe 沙箱  | ✅ 自动生效                |
| CSS 变量桥接   | 主应用定义变量，子应用消费 | 两者皆可    | ✅ 实现主题适配            |

### 6.3 Dark Mode 同步策略

主应用 `watch(themeStore.mode)` → `microApp.setData()` 推送 `{ isDark }` → 子应用 `addDataListener` 响应 → `classList.toggle('dark')`。

**关键点**：主题同步通过 `setData` 精准推送，不触发子应用重渲染，性能开销极低。

## 七、性能优化策略

### 7.1 预加载（preFetch）

官方提供 `preFetch` API，可在空闲时预加载子应用资源，**首次切换提速 60%+**：

```typescript
import microApp from '@micro-zoe/micro-app'

// 在主应用启动后预加载
microApp.preFetch([
  { name: 'logistics', url: 'http://localhost:3003' },
  { name: 'warehouse', url: 'http://localhost:3004' },
])
````

> **当前状态**：⬜ 未实现。建议在 `setupMicroApp()` 中根据 `getAllMicroApps()` 自动预加载。

### 7.2 Keep-Alive 保活

已通过 `<micro-app keep-alive>` 实现，子应用切换时不会销毁，再次打开瞬间恢复。

**注意事项**：

- keep-alive 会增加内存占用，高频使用的 3-5 个应用适合保活
- 低频应用建议设置 `destroy` 属性，卸载时释放资源
- 可通过 `clear-data` 属性在卸载时清空通信数据

### 7.3 Fiber 异步渲染

官方提供 `fiber` 模式，将 JS 执行拆分为多个微任务，**避免长任务阻塞主线程**：

```typescript
// 全局开启
microApp.start({ fiber: true })

// 或单应用开启
<micro-app name="heavy-app" url="..." fiber />
```

> **适用场景**：子应用 JS 体积大（>1MB）、首次渲染耗时长的场景。

### 7.4 全局资源共享（globalAssets）

将 Vue、Naive UI 等公共依赖配置为全局资源，**所有子应用共享，减少重复加载**：

```typescript
microApp.start({
  globalAssets: {
    js: ['https://cdn.example.com/vue@3.5.13/vue.global.prod.js'],
    css: ['https://cdn.example.com/common-styles.css'],
  },
})
```

> **适用场景**：多个子应用使用相同版本的 Vue/Naive UI 时，通过 CDN 共享可减少 30%+ 总包体积。

### 7.5 性能优化清单

| 优化项                       | 预期收益          | 优先级 | 状态      |
| ---------------------------- | ----------------- | ------ | --------- |
| 子应用预加载（preFetch）     | 首次切换提速 60%  | P0     | ⬜ 待实现 |
| Keep-Alive 保活              | 切换零延迟        | P0     | ✅ 已实现 |
| Fiber 异步渲染               | 避免长任务阻塞    | P1     | ⬜ 待实现 |
| 共享依赖 CDN（globalAssets） | 减少 30% 总包体积 | P1     | ⬜ 待实现 |
| 子应用懒加载路由             | 减少子应用 FCP    | P1     | ⬜ 待实现 |
| 静态资源 CDN 部署            | 加载速度 2-3x     | P2     | ⬜ 待实现 |
| 子应用资源缓存策略           | 二次加载提速      | P2     | ⬜ 待实现 |

## 八、团队协作模式

### 8.1 独立开发模式

```
             ┌── 主应用团队 ── Robot Admin（门户 + 容器）
             │
项目组 ──────┼── 团队 A ────── 物流子应用（独立仓库/独立部署）
             │
             └── 团队 B ────── 仓储子应用（独立仓库/独立部署）
```

**协作约定**：

1. 主应用团队维护 `shared/` 契约层和《子应用接入规范》
2. 子应用团队消费 `@shared/*` 类型定义和常量
3. 子应用团队独立开发、独立测试、独立部署
4. 联调时使用 `.env.local` 配置对方的开发地址

### 8.2 Monorepo 统一管理模式

适合团队规模 <5 人、所有子应用归同一团队管理的场景。

```
Robot_Admin/
├── src/                      # 主应用
├── shared/                   # 契约层
├── micro-apps/
│   ├── logistics/            # 子应用 A
│   └── warehouse/            # 子应用 B
└── package.json              # 统一命令: dev:micro / build:all
```

### 10.3 错误边界增强建议

当前 `@error` 仅展示 NResult，生产环境建议：

- **自动重试**：最多 3 次，指数退避（1s → 2s → 4s）
- **超时检测**：10s 未加载完成自动触发降级
- **降级方案**：iframe 直接嵌入子应用 URL
- **错误上报**：接入 Sentry / 自研监控平台

### 10.4 PostMessage 安全增强
